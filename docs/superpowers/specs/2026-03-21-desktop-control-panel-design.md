# 轻量级 HIS 原生桌面控制页设计

## 1. 目标

在当前 `C` 语言 HIS 项目上增加一个 `原生桌面控制页`，用于替代部分控制台交互，让系统具备更直观的登录、总览和业务操作入口。

本轮只做 `MVP`，范围限定为：

- 登录
- 首页仪表盘
- 患者查询
- 挂号录入
- 发药记录查询

实现约束：

- 必须使用 `纯 C`
- 必须是 `原生桌面控制页`
- GUI 方案采用 `raylib + raygui`

## 2. 设计结论

### 2.1 总体方案

采用 `单窗口侧边栏工作台` 结构：

- 左侧导航栏
- 顶部状态栏
- 右侧主工作区
- 页内消息条

不采用多窗口、多弹窗、多标签复杂布局，避免在 `raylib + raygui` 下引入过高状态复杂度。

### 2.2 页面结构

左侧导航固定为：

- 首页
- 患者
- 挂号
- 发药记录
- 系统

顶部状态栏固定显示：

- 系统标题
- 当前登录用户
- 当前角色
- 当前时间
- 退出按钮

右侧工作区按导航切换页面内容。

## 3. 视觉方向

### 3.1 视觉语言

采用 `医疗后台 + 轻玻璃感` 风格，目标是：

- 干净
- 稳定
- 信息清晰
- 易扩展

不做夸张大屏或重动画设计。

### 3.2 色彩

- 主色：`#2563EB`
- 次色：`#3B82F6`
- 强调色：`#F97316`
- 背景：`#F8FAFC`
- 面板：`#FFFFFF`
- 主文本：`#1E293B`
- 次文本：`#64748B`
- 成功：`#16A34A`
- 警告：`#D97706`
- 错误：`#DC2626`

### 3.3 控件规则

- 导航按钮：大点击区，选中态高亮
- 卡片：轻阴影、浅边框、固定圆角
- 表格：轻量行列表 + 选中详情
- 表单：标签在上，输入框统一宽度
- 消息条：页内顶部提示，统一成功/失败反馈

## 4. 页面设计

### 4.1 登录页

功能：

- 输入用户编号
- 输入密码
- 选择角色或由账号角色自动决定
- 登录按钮

行为：

- 登录失败时，在页面顶部显示错误提示
- 登录成功后进入主工作台
- 患者登录后，自动绑定患者会话

### 4.2 首页仪表盘

展示内容：

- 患者总数
- 今日挂号数
- 当前住院数
- 低库存药品数

下方展示：

- 最近挂号记录
- 最近发药记录

首版数据允许采用：

- 页面进入时聚合一次
- 点击刷新按钮重新加载

### 4.3 患者页

布局：

- 顶部搜索栏
- 中间结果列表
- 下方或右侧详情面板

查询维度：

- 患者编号
- 患者姓名

详情展示：

- 基本信息
- 联系方式
- 过敏史
- 病史
- 当前住院状态

### 4.4 挂号页

布局：

- 左侧表单
- 右侧最近挂号记录/提交反馈

录入字段：

- 患者编号
- 医生编号
- 科室编号
- 挂号时间

行为：

- 提交后调用现有 `RegistrationService`
- 成功后刷新最近挂号列表
- 失败后保留表单内容并提示错误

### 4.5 发药记录页

布局：

- 顶部筛选区
- 中间发药记录表格
- 底部详情或状态栏

首版查询方式：

- 患者编号查询

显示字段：

- 发药编号
- 患者编号
- 处方编号
- 药品编号
- 数量
- 发药人
- 发药时间

患者登录时，该页默认只显示当前患者记录。

## 5. 代码结构设计

### 5.1 模块拆分

新增桌面 UI 建议拆为以下模块：

- `include/ui/DesktopApp.h`
- `src/ui/DesktopApp.c`
- `include/ui/DesktopTheme.h`
- `src/ui/DesktopTheme.c`
- `include/ui/DesktopPages.h`
- `src/ui/DesktopPages.c`
- `include/ui/DesktopAdapters.h`
- `src/ui/DesktopAdapters.c`

### 5.2 职责划分

#### `DesktopApp`

负责：

- 窗口生命周期
- 页面切换
- 全局 session
- 顶部状态栏
- 左侧导航
- 主循环

#### `DesktopTheme`

负责：

- 颜色
- 间距
- 字号
- 卡片样式
- 状态色

#### `DesktopPages`

负责：

- 登录页绘制
- 首页绘制
- 患者页绘制
- 挂号页绘制
- 发药记录页绘制

#### `DesktopAdapters`

负责：

- 桌面页状态与现有 service 层之间的调用转换
- 查询结果转换为桌面页可渲染结构
- 统一错误消息格式

## 6. 数据流

### 6.1 登录流

1. 启动程序
2. 显示登录页
3. 用户输入账号和密码
4. 调用现有 `AuthService`
5. 成功后写入全局 session
6. 若为患者角色，同步绑定患者会话
7. 进入首页

### 6.2 页面业务流

统一规则：

- 页面只维护 UI 状态
- 页面不直接读写 txt 文件
- 所有业务调用都经过现有 service 或 adapter
- service 返回 `Result` 后：
  - 成功：刷新页面数据
  - 失败：显示消息条

### 6.3 会话规则

全局 session 至少包含：

- 登录用户编号
- 登录角色
- 患者绑定编号（如果当前是患者）
- 最近一次操作消息

## 7. raylib + raygui 适配策略

### 7.1 适合做的内容

- 单窗口后台界面
- 按钮、输入框、标签、面板
- 轻量表格列表
- 卡片布局
- 导航和状态栏

### 7.2 本轮不做的内容

- 多窗口停靠
- 富文本编辑器
- 复杂可排序表格
- 多层模态交互系统
- 动态图表动画

### 7.3 实现原则

- 先保证稳定布局和交互清晰
- 不为视觉效果引入复杂渲染技巧
- 尽量使用简单矩形、文本、边框和状态色构建页面

## 8. 错误处理

统一错误处理规则：

- 输入缺失：页内即时提示
- 查询失败：显示失败消息条
- 提交失败：保留用户输入
- 数据为空：显示空状态，不显示空白页

长输入、无效数字等底层校验继续复用已有逻辑。

## 9. 测试策略

### 9.1 保持现有业务测试

现有 `CTest` 业务测试不回退，不拆除。

### 9.2 新增桌面层测试重点

优先测试非图形逻辑：

- 页面状态切换
- 登录状态切换
- 查询条件校验
- 消息条状态变化
- adapter 与 service 之间的参数映射

### 9.3 图形验证

本轮图形层采用：

- 构建通过
- 应用可启动
- 登录可进入首页
- 页面切换和核心交互 smoke 测试

不要求首轮就做完整自动化 GUI 测试。

## 10. 推荐实现顺序

1. 接入 `raylib + raygui`
2. 建立 `DesktopApp` 和 `DesktopTheme`
3. 完成登录页与 session
4. 完成首页仪表盘
5. 完成患者页
6. 完成挂号页
7. 完成发药记录页
8. 做启动与交互 smoke

## 11. 本轮不扩展内容

以下内容明确不纳入本轮：

- 医生看诊页
- 住院/床位控制页
- 药房发药操作页
- 报表图表系统
- 多角色复杂权限矩阵
- 自定义主题切换

## 12. 结论

本方案以 `raylib + raygui` 为基础，在不破坏现有纯 C HIS 结构的前提下，增加一个 `单窗口原生桌面控制页`。

它的价值在于：

- 让当前控制台系统拥有可演示、可扩展的桌面入口
- 保持业务逻辑继续复用现有 service 层
- 把首版复杂度控制在 `登录 + 仪表盘 + 患者查询 + 挂号 + 发药记录查询`

该方案适合作为当前项目的下一步 UI 增量，而不是一次性重做整套系统。
