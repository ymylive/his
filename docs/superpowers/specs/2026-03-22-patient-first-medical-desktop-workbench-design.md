# 患者优先的原生桌面医疗工作台设计

## 1. 背景

当前项目已经具备较完整的 HIS 业务能力：

- 患者信息管理
- 医生与科室管理
- 挂号、接诊、检查
- 住院、病房、床位
- 药房、库存、发药
- 七类角色账号认证

当前桌面端已经基于 `C + raylib + raygui` 落地了登录、仪表盘、患者、挂号、医生、住院、药房、发药记录等页面骨架，但整体仍是 `MVP`：

- 页面更像业务表单集合，不像真正的工作台
- 多数页面仍以“左侧输入 + 右侧文本输出”为主
- 已有业务能力没有完整映射到桌面端
- 角色之间缺少清晰的主任务分化
- 视觉风格仍停留在“课程设计后台原型”

本设计的目标是在 **不改变技术栈、不重写业务层、不引入 Web 前端** 的前提下，把桌面端升级成真正可用的多角色医疗工作台。

## 2. 目标与约束

### 2.1 目标

- 完成原生桌面端对现有 HIS 核心功能的完整覆盖
- 保持 `纯 C`、`原生桌面`、`单窗口工作台` 路线
- 建立统一的医疗行业视觉语言
- 用“患者优先”的工作台体验作为整套系统的母版
- 让其它角色在统一壳内形成各自清晰的专业工作台
- 保证中文稳定显示，无乱码

### 2.2 约束

- 只使用现有项目中的 `C / CMake / raylib / raygui`
- 业务调用必须继续复用现有 `MenuApplication / service / repository`
- 桌面层不直接读写 `txt` 文件
- 交互必须稳定，不依赖脆弱的临时焦点状态
- 改造后仍需保持桌面 smoke 与既有测试可验证

### 2.3 非目标

以下内容明确不纳入本轮：

- 收费、缴费、结算
- 真实号源排班系统
- 多窗口、多标签停靠布局
- 富文本病历编辑器
- 复杂图表大屏
- 与现有业务无关的大规模重构

### 2.4 分阶段交付与切线

本规格描述的是 **目标态蓝图**，但后续实施不能把全部内容塞进一个实现计划中。

后续必须按阶段拆分：

#### Phase 1：共享壳与患者薄切片

必须包含：

- 统一桌面壳
- 统一主题与公共组件
- 患者工作台完整落地
- 患者自助挂号、病历检查、住院状态、发药用药、我的信息
- 必要的桌面聚合查询与刷新机制

不包含：

- 管理员跨域维护大屏
- 所有角色的完整深度功能

#### Phase 2：挂号员与医生

必须包含：

- 挂号员接待首页与患者维护、挂号办理、查询/取消
- 医生待诊工作台、患者历史、接诊记录、检查处理、库存参考

#### Phase 3：住院登记员、病区管理员、药房

必须包含：

- 住院登记员入出院工作台
- 病区管理员病区总览、床位管理、转床、出院前检查
- 药房发药工作台、库存、入库、低库存、发药记录

#### Phase 4：系统管理员与统一打磨

必须包含：

- 管理员运营总览
- 患者、医生科室、记录查询、病房床位、药品总览桌面化
- 统一视觉与交互打磨
- 交叉角色联动验证

后续实现计划要求：

- 一次只为一个阶段出详细实施计划
- 当前最先进入详细规划的是 `Phase 1`
- 后续阶段在前一阶段交付可验证后再展开

## 3. 方案比较与选型

本轮比较了三种信息架构方案：

### 3.1 方案 A：患者中心统一工作台

所有角色尽量共用同一套导航结构，只按权限显示局部差异。

优点：

- 结构最统一
- 改造成本最低

缺点：

- 医生、药房、住院等角色会缺乏专业工作台感
- 容易变成“一套页面给所有人换按钮”

### 3.2 方案 B：完全独立角色工作站

每个角色拥有完全不同的首页与导航。

优点：

- 角色感最强
- 最接近正式医院软件

缺点：

- 工作量过大
- 风格容易分裂
- 当前桌面页代码会快速失控

### 3.3 方案 C：统一外壳 + 角色专属主工作台

保留统一的桌面壳、组件系统和视觉语言；每个角色登录后进入自己的主工作台。

优点：

- 兼顾统一性与专业性
- 与当前单窗口架构最匹配
- 便于从现有桌面 MVP 平滑升级

缺点：

- 需要补一层首页与公共组件抽象

### 3.4 结论

本轮采用 **方案 C**。

它允许系统保持统一原生桌面外壳，同时让患者、挂号员、医生、住院、病区、药房、管理员形成各自清晰的主任务入口。

## 4. 设计原则

### 4.1 患者优先

患者工作台是整套系统的体验母版。原因不是“患者比其它角色更重要”，而是患者视角最能把挂号、就诊、检查、住院、发药主链路串起来。

### 4.2 工作台优先

每个角色首页必须是“当前最需要处理的任务台”，而不是静态菜单页。

### 4.3 医疗行业视觉

整套系统采用温和、明亮、可信的医疗视觉风格，不使用金融深色盘、赛博风或夸张渐变。

### 4.4 能力真实

只表达当前后端真实支持的业务能力，不伪造收费、真实号源、完整处方闭环等系统并不存在的能力。

### 4.5 稳定优先

优先保证：

- 中文显示稳定
- 交互清晰
- 状态同步正确
- 页面结构可维护

## 6. 患者工作台设计

患者工作台是本轮的主母版。总体定位为：

- 就诊全流程型
- 当前流程看板型首页
- 一页式主流程办理
- 历史记录按功能分类

## 6.1 首页流程

首页结构分为四块：

### 顶部欢迎区

显示：

- 患者姓名 / 编号
- 当前主状态
- 关键提醒摘要

### 当前流程主看板

使用四段式流程卡：

- 待挂号
- 待就诊
- 待检查 / 待结果
- 待取药 / 住院中

规则：

- 始终只高亮当前最需要处理的一步
- 每一步都显示简短状态说明
- 当前步骤下直接给出下一步动作

### 6.1.1 当前主状态判定矩阵

患者首页只能高亮一个“当前主状态”，必须按以下优先级判定：

| 优先级 | 条件 | 当前主状态 | 高亮步骤 | 下一步动作 |
| --- | --- | --- | --- | --- |
| 1 | 存在 `Admission.status == ACTIVE` 的住院记录 | 住院中 | 第 4 步 | 查看住院状态 |
| 2 | 存在 `Registration.status == PENDING` 的挂号记录 | 待就诊 | 第 2 步 | 查看挂号候诊 |
| 3 | 不存在活动住院与待诊挂号，但存在 `ExaminationRecord.status == PENDING` | 待检查/待结果 | 第 3 步 | 查看病历检查 |
| 4 | 不存在以上状态，但最近一次就诊后存在已完成发药记录 | 已发药 | 第 4 步 | 查看发药用药 |
| 5 | 不存在以上状态，但最近一次挂号已诊毕或最近一次就诊已完成 | 已完成本次就诊 | 第 3 步 | 查看病历检查 |
| 6 | 不存在任何未取消挂号、就诊、检查、住院、发药记录 | 待挂号 | 第 1 步 | 去挂号 |

补充规则：

- `待处理挂号` 的确定条件为：存在任意一条当前患者的 `Registration.status == REG_STATUS_PENDING`
- 由于当前业务层只持久化已完成的发药记录，不从桌面端伪造“待取药队列”
- 若同时存在住院和待诊挂号，以住院中为最高优先级，因为它代表当前最强约束状态

### 6.1.2 最近流程选择规则

首页下方“最近流程列表”必须按以下规则生成：

1. 以当前患者最近 `3` 条 `未取消挂号记录` 作为流程基座，按 `registered_at` 倒序排序
2. 每条流程通过 `registration_id -> visit_id -> examination_records` 关联出就诊与检查状态
3. 住院与发药信息不单独生成新流程行，只作为该患者当前全局状态标签附着在列表行右侧
4. 若患者没有任何未取消挂号记录，则列表为空并显示“去挂号”空状态

### 6.1.3 关键提醒生成规则

首页欢迎区的提醒按以下条件生成，最多展示 `3` 条，优先级从高到低：

1. 当前住院中
2. 存在待诊挂号
3. 存在待完成检查
4. 最近一次发药已完成
5. 最近一次就诊已完成但无其它待处理状态

同类提醒只显示一条；较低优先级提醒在高优先级提醒占满时不展示。

### 一页式挂号办理区

布局：

- 左侧：科室列表
- 中间：医生列表
- 右侧：当前选择与确认卡

规则：

- 先选科室，再选医生
- 不要求患者填写挂号时间
- 直接使用当前时间完成快捷挂号
- 若已有待处理挂号，则挂号区切换为“当前挂号摘要 + 取消入口”

### 最近流程列表

展示最近 `3` 条流程，每条流程用紧凑状态摘要展示：

- 挂号
- 看诊
- 检查
- 住院
- 发药

点击某条记录后，在详情区展示该流程摘要。

## 6.2 挂号候诊

负责患者的挂号与当前候诊信息。

包含：

- 科室选择
- 医生选择
- 当前挂号状态
- 挂号历史
- 挂号取消

## 6.3 病历检查

负责患者的病历与检查视图。

包含：

- 最近就诊摘要
- 历次看诊记录
- 检查申请列表
- 检查结果列表

## 6.4 住院状态

负责患者住院信息展示。

包含：

- 当前是否住院
- 病房 / 床位
- 当前住院摘要
- 历史住院记录

## 6.5 发药用药

负责患者的发药和药品说明。

包含：

- 最近发药记录
- 历史发药记录
- 药品说明 / 用药方法

## 6.6 我的信息

负责患者静态档案展示。

包含：

- 基本资料
- 联系方式
- 过敏史
- 病史
- 当前住院标记

## 7. 其它角色工作台分化

## 7.1 挂号员

首页定位为 `接待首页`，强调接待与事务办理。

首页内容：

- 今日接待量
- 今日挂号量
- 最近挂号记录
- 患者快速检索
- 快捷挂号办理

子页职责：

- 患者检索：快速查询患者信息
- 挂号办理：完整录入与确认
- 挂号查询：挂号查询与取消
- 患者维护：新增、修改、查询

首页布局规则：

- 左侧固定为患者快速检索与最近接待列表
- 中间为挂号办理主表单
- 右侧为当前选中患者详情、最近挂号结果与取消入口

刷新与异常规则：

- 挂号成功后刷新接待首页、挂号查询页、相关患者摘要
- 挂号失败时保留患者与医生选择
- 无查询结果时显示“可先新增患者或更换查询条件”

## 7.2 医生

首页定位为 `待诊工作台`，强调待诊与病历处理。

首页内容：

- 当前待诊数量
- 今日已接诊数量
- 待回写检查
- 待诊队列
- 当前患者摘要
- 快捷操作区

子页职责：

- 患者历史：病史与历史记录查询
- 接诊记录：新增诊疗记录
- 检查处理：新增检查、回写结果
- 用药参考：库存查询与用药辅助参考

注意：

- 不伪装成完整处方执行系统
- 只暴露当前后端实际支持的诊疗、检查、库存相关能力

首页布局规则：

- 左侧为待诊队列
- 中间为当前患者摘要与历史摘要
- 右侧为接诊、开检查、回写检查结果、查库存的快捷操作卡

刷新与异常规则：

- 接诊成功后刷新待诊队列与当前患者摘要
- 检查新增或回写成功后刷新检查处理页与患者病历检查页
- 查询为空时明确显示“当前无待诊患者”或“无历史记录”

## 7.3 住院登记员

首页定位为 `入出院工作台`，强调入出院事务。

首页内容：

- 当前在院人数
- 今日入院
- 今日出院
- 空床摘要
- 入院办理入口
- 出院办理入口

子页职责：

- 床位查询
- 入院办理
- 出院办理
- 在院查询

首页布局规则：

- 顶部指标卡显示在院、今日入院、今日出院、空床
- 左侧为入院办理
- 中间为出院办理
- 右侧为在院查询与床位查询结果

刷新与异常规则：

- 入院/出院成功后，首页指标、床位查询和患者住院状态同步失效并重载
- 床位占用冲突时保留当前表单并提示真实冲突原因

## 7.4 病区管理员

首页定位为 `病区总览`，强调病房资源与床位状态。

首页内容：

- 病房占用率
- 空床数
- 待调整床位
- 待出院检查
- 病房列表
- 床位状态

子页职责：

- 病房列表
- 床位管理
- 住院患者查询
- 转床与出院前检查

首页布局规则：

- 左侧为病房与床位总览
- 中间为当前病房 / 床位详情
- 右侧为转床与出院前检查动作区

刷新与异常规则：

- 转床成功后刷新病房列表、床位状态和患者住院详情
- 出院前检查失败时保留选中住院记录，不清空上下文
- 空病房、空床位必须显示明确空状态，不允许纯空白区域

## 7.5 药房

首页定位为 `发药工作台`，强调发药与库存。

首页内容：

- 低库存药品数
- 今日发药数
- 最近库存动作
- 发药操作
- 库存查询
- 低库存提醒

子页职责：

- 药品库存
- 药品入库
- 低库存提醒
- 发药记录

首页布局规则：

- 左侧为发药操作
- 中间为库存查询与入库
- 右侧为低库存提醒与最近发药结果

刷新与异常规则：

- 发药成功后刷新库存、低库存提醒、患者发药页
- 入库成功后刷新库存与低库存提醒
- 库存不足冲突时保留当前发药表单并显示真实库存结果

## 7.6 系统管理员

首页定位为 `运营总览`，强调跨域监控与维护。

首页内容：

- 患者总数
- 挂号总数
- 当前住院数
- 低库存药品数
- 最近关键记录摘要

子页职责：

- 患者维护
- 医生科室
- 记录查询
- 病房床位
- 药品总览

管理员必须正式获得桌面端入口的能力包括：

- 患者维护
- 科室维护
- 医生维护 / 按科室查看医生
- 按时间范围查询医疗记录
- 病房 / 床位总览
- 药品库存 / 低库存总览

首页布局规则：

- 顶部为跨域指标卡
- 中间为最近关键记录摘要
- 下方为五个管理域入口卡：患者、医生科室、记录、病房床位、药品

刷新与异常规则：

- 任一管理动作成功后，仅使相关域失效，不强制整页重载
- 跨域数据刷新失败时保留最后一次成功快照，并显示“数据已过期，可手动刷新”

## 8. 视觉系统

## 8.1 整体气质

- 洁净

- 亲和
- 稳定

不采用：

- 金融深色盘
- 赛博科技风
- 重玻璃态
- 夸张渐变

- 主背景：`#F4FAF9`

- 次级面板：`#EAF6F5`
- 主品牌色：`#0F9FA8`
- 次品牌色：`#3BB8C4`
- 主文本：`#16343A`
- 次文本：`#5F7D83`
- 边框：`#CFE5E4`
- 成功：`#22A06B`
- 警告：`#D89B2B`
- 错误：`#D95C5C`
- 当前流程高亮：`#E0F7F5`
- 处理中高亮：`#E8F4FF`

## 8.3 字体与中文

继续使用系统中文字体优先加载策略，保证：

- 中文稳定显示
- 标题和正文都可读
- 无乱码

不引入需要额外打包的高风险字体依赖。

## 8.4 组件规则

统一组件包括：

- 导航按钮
- 指标卡
- 流程卡
- 表单卡
- 列表卡
- 详情卡
- 状态标签
- 空状态面板
- 结果摘要卡

规则：

- 标签始终在输入框上方
- 输入点击区不小于 `44px`
- 成功/失败反馈统一走页内消息条
- 长结果进入明细区，不塞给消息条
- 所有空状态必须带说明与下一步提示

## 8.5 布局规范

- 顶栏轻量化，不抢主视图
- 侧栏使用浅色医疗导航风格
- 页面主区优先采用“概览 + 主操作 + 详情/结果”
- 默认适配 `1600x960`
- 小窗口下仍保持无明显横向崩坏

## 9. 数据流与交互规则

## 9.1 登录与会话

- 登录页保留账号、密码、角色辅助选择
- 实际进入角色以认证结果为准
- 患者登录后自动绑定 `patient_id`
- 医生登录后默认带入 `doctor_id`
- 药房登录后默认带入 `pharmacist_id`
- 退出登录后清空所有页面状态

## 9.2 患者自助挂号

- 患者可以直接自助挂号
- 挂号流程为：科室 -> 医生 -> 确认
- 挂号时间由系统自动取当前时间
- 若已有未完成挂号，则前端不再鼓励重复挂号
- 挂号失败时保留当前选择
- 挂号成功后立即刷新患者首页与挂号状态

补充规则：

- “已有未完成挂号”的唯一判定条件是存在任意 `REG_STATUS_PENDING` 挂号
- 若存在未完成挂号，首页挂号区切换为只读摘要，不允许再次提交
- 患者主动取消挂号后，首页流程重新按状态矩阵计算

## 9.3 页面刷新

页面进入时的规则：

- 首页类页面首次进入自动加载
- 查询页默认保留上次上下文

统一失效模型：

- 每个页面都维护 `loaded / dirty / stale` 三态
- 关键动作成功后，不直接强制清空页面，而是把依赖页面标记为 `dirty`
- 可见页面在下一次绘制前自动重载 `dirty` 数据
- 若自动重载失败，则页面进入 `stale`，保留最后一次成功快照，并显示警告条

关键动作的失效矩阵：

| 动作 | 失效页面 | 成功后行为 | 重载失败时 | 冲突/陈旧数据策略 |
| --- | --- | --- | --- | --- |
| 登录成功 | 当前角色首页、全局摘要 | 自动加载首页 | 保留空页面并显示初始化失败 | 不允许进入非当前角色页 |
| 患者挂号 | 患者首页、挂号候诊、挂号员接待首页 | 自动重载相关摘要 | 保留成功消息，页面标记 stale | 以后端挂号状态为准 |
| 挂号取消 | 患者首页、挂号候诊、挂号查询 | 自动重载当前挂号状态 | 保留取消结果与旧快照 | 若记录已诊毕/已取消，则以后端失败结果为准 |
| 接诊记录提交 | 医生待诊、患者病历检查、患者首页状态 | 自动重载待诊与病历摘要 | 保留表单结果，摘要标记 stale | 若挂号状态已变化，以后端返回为准 |
| 检查新增/回写 | 医生检查处理、患者病历检查、患者首页状态 | 自动重载检查摘要 | 保留当前选中记录与旧快照 | 若检查状态已被他人更新，以后端状态为准 |
| 入院/出院/转床 | 住院登记员首页、病区总览、患者住院状态 | 自动重载住院与床位摘要 | 保留动作结果，住院视图标记 stale | 床位占用、住院状态冲突以床位/住院后端状态为准 |
| 发药/入库 | 药房首页、患者发药用药、管理员药品总览 | 自动重载库存与发药摘要 | 保留动作结果，库存视图标记 stale | 库存不足或并发入库后以库存后端状态为准 |
| 患者/医生/科室维护 | 对应管理页与关联首页摘要 | 自动重载当前管理域 | 保留编辑上下文，管理域标记 stale | 若对象已被删除或更新，以后端状态为准 |

关键动作完成后的规则：

- 挂号成功：刷新患者首页、挂号页、相关概览
- 医生完成接诊或检查操作：刷新医生工作台、患者病历检查页
- 入院/出院成功：刷新住院页、病区页、患者住院状态
- 发药成功：刷新药房库存、患者发药页、概览提醒

## 9.4 结果与反馈

- 成功：明确写出结果与编号
- 失败：明确写出失败原因
- 警告：用于当前状态不允许继续
- 消息条只保留最近一次关键反馈
- 大段文本只作为补充输出，不作为主要展示方式

## 9.5 权限规则

- 患者仅可查看自己的记录
- 医生默认只看自己的待诊与工作数据
- 业务角色仅展示与岗位匹配的入口
- 管理员可跨域总览，但不需要伪装成业务角色首页

## 9.6 空状态

每类空状态都必须明确说明当前无数据并给出下一步：

- 无挂号：提示去挂号
- 无检查：提示暂无检查记录
- 无住院：提示当前未住院
- 无发药：提示暂无发药记录

## 10. 架构与模块边界

桌面层必须继续以清晰边界组织，避免把所有逻辑塞进单个 `DesktopPages.c`。

## 10.1 核心单元

核心单元必须按接口契约组织，而不是只按“职责描述”组织。

| 单元 | Owned State | 输入 | 输出 | 依赖 | 测试缝 | 备注 |
| --- | --- | --- | --- | --- | --- | --- |
| `DesktopApp` | 当前用户、当前页面、全局消息、页面失效标记 | 登录结果、导航事件、窗口事件 | 页面切换、会话清理、全局消息 | `DesktopTheme`、页面控制器 | `test_desktop_state` | 不直接做业务查询 |
| `DesktopTheme` | 颜色令牌、间距、字体句柄 | 主题常量、系统字体路径 | raygui style、文本绘制参数 | raylib/raygui | 主题 smoke + 字体回退验证 | 只负责视觉，不存业务状态 |
| `DesktopWidgets` | 无业务状态，仅可选局部 UI 状态 | 视图模型、绘制参数 | 卡片、流程条、列表、详情、空状态 | `DesktopTheme` | 通过页面集成验证 | 本轮从“建议”升级为必须抽出公共组件 |
| `DesktopAdapters` | 页面聚合查询缓存键、结构化结果 | 页面动作参数 | 页面视图模型、统一 `Result` | `MenuApplication` | `test_desktop_adapters` | 桌面层唯一业务桥接点 |
| `PatientWorkbenchController` | 患者首页、挂号、病历、住院、发药页面状态 | 用户事件、适配层结果 | 患者工作台绘制模型 | `DesktopAdapters` | `test_desktop_workflows` | 患者母版 |
| `ClerkWorkbenchController` | 接待首页、挂号办理、患者维护状态 | 用户事件、适配层结果 | 挂号员工作台绘制模型 | `DesktopAdapters` | `test_desktop_workflows` | 面向挂号员 |
| `DoctorWorkbenchController` | 待诊、历史、接诊、检查、库存状态 | 用户事件、适配层结果 | 医生工作台绘制模型 | `DesktopAdapters` | `test_desktop_workflows` | 面向医生 |
| `InpatientWorkbenchController` | 入出院、病区、床位状态 | 用户事件、适配层结果 | 住院/病区绘制模型 | `DesktopAdapters` | `test_desktop_workflows` | 可按登记员/病区管理员拆子视图 |
| `PharmacyWorkbenchController` | 发药、库存、入库、低库存状态 | 用户事件、适配层结果 | 药房工作台绘制模型 | `DesktopAdapters` | `test_desktop_workflows` | 面向药房 |
| `AdminWorkbenchController` | 运营总览与跨域管理状态 | 用户事件、适配层结果 | 管理员工作台绘制模型 | `DesktopAdapters` | `test_desktop_workflows` | Phase 4 落地 |

文件组织要求：

- `DesktopPages.c` 不再继续单文件膨胀
- 必须按角色或页面域拆成可独立理解的渲染单元
- 所有页面控制器只拥有自己的页面状态
- 所有业务调用统一经 `DesktopAdapters`
- 页面控制器不得直接访问 `repository`

## 10.2 需要抬升到桌面层的能力

桌面层必须明确区分三类能力：

| 能力 | 当前业务层状态 | 桌面层动作 | 备注 |
| --- | --- | --- | --- |
| 患者新增 / 修改 / 查询 | 已有公开 `MenuApplication_*` | 直接做适配器封装 | 已有公共接口 |
| 挂号新增 / 查询 / 取消 | 已有公开 `MenuApplication_*` | 直接做适配器封装 | 已有公共接口 |
| 待诊 / 病史 / 接诊 / 检查 | 已有公开 `MenuApplication_*` | 直接做适配器封装 | 已有公共接口 |
| 病房 / 床位 / 入院 / 出院 / 当前住院 | 已有公开 `MenuApplication_*` | 直接做适配器封装 | 已有公共接口 |
| 药品新增 / 入库 / 发药 / 查库存 / 低库存 | 已有公开 `MenuApplication_*` | 直接做适配器封装 | 已有公共接口 |
| 患者删除 | 仅 `MenuApplication.c` 内部静态函数存在 | 提升为公开应用层能力或新增 `DesktopAdapters` 专用公开桥接 | 管理员/挂号员维护需要 |
| 科室新增 / 修改 | 仅内部静态函数存在 | 提升为公开应用层能力 | 管理员需要 |
| 医生新增 / 查询 / 按科室查看 | 部分仅内部静态函数存在 | 提升为公开应用层能力 | 管理员需要 |
| 按时间范围查询记录 | 仅内部静态函数存在 | 提升为公开应用层能力 | 管理员记录查询需要 |
| 转床 | 仅内部静态函数存在 | 提升为公开应用层能力 | 病区管理员需要 |
| 出院前检查 | 仅内部静态函数存在 | 提升为公开应用层能力 | 病区管理员需要 |
| 药品使用方法查询 | 仅内部静态函数存在 | 提升为公开应用层能力 | 患者发药用药页需要 |
| 患者首页聚合摘要 | 业务层无单一接口 | 新增 `DesktopAdapters` 聚合查询 | 桌面专用聚合，不回写业务规则 |

原则：

- 已有公开接口的能力只做适配，不重写业务逻辑
- 仅内部静态存在的能力，必须先提升到可测试的公开应用层，再给桌面使用
- 桌面聚合查询只负责组合展示，不新增业务真相

## 11. 错误处理与边界情况

必须明确处理以下场景：

- 登录失败
- 角色选择与真实角色不一致
- 患者编号不存在
- 医生或科室编号无效
- 床位已占用
- 住院记录不存在或已结束
- 药品库存不足
- 当前患者无相关记录
- 查询结果为空
- 当前角色无权访问目标页面

所有失败都必须：

- 保留必要上下文
- 不让页面进入不可恢复状态
- 用明确消息反馈原因

## 12. 测试与验收

## 12.1 自动验证

至少覆盖：

- 桌面状态切换测试
- 桌面适配层测试
- 角色工作流测试
- smoke 启动测试

## 12.2 人工验证

需要确认：

- 中文稳定显示
- 所有角色可登录
- 所有角色进入正确默认首页
- 患者流程主链路可演示
- 关键操作后页面状态会刷新
- 无明显空白页与乱码
- `1600x960` 下无布局遮挡
- `1366x768` 下主区无不可操作控件
- 不出现明显横向裁切导致的信息不可读

## 12.3 通过标准

交付完成时必须满足：

- 七类角色都有可用主工作台
- 患者端主链路完整可用
- 桌面端覆盖现有核心业务能力
- 风格统一，符合明亮医疗桌面方向
- 构建通过
- 相关测试通过
- 桌面 smoke 通过
- `1600x960` 为主验收分辨率，`1366x768` 为最小支持分辨率
- 所有关键按钮点击区不小于 `44px`
- 中文文本在主流程页面不得出现系统性乱码或成片截断

## 13. 风险与延后项

本轮允许明确延后的内容：

- 真实排班与号源系统
- 复杂收费系统
- 电子处方执行闭环
- 高级报表图表

这些延后项必须在文档和实现中保持诚实，不做伪入口或伪流程。

## 14. 结论

本设计选择在现有 `C + raylib + raygui` 原生桌面架构上，建设一套 **患者优先、角色分化、风格统一、能力真实** 的医疗工作台。

它不是对现有系统的推倒重做，而是一次有边界的桌面前端升级：

- 用患者工作台定义主体验
- 用角色工作台完成专业分化
- 用统一视觉系统建立正式软件观感
- 用适配层扩展把已有业务能力真正接到桌面端

该设计完成后，项目将从“桌面 MVP 演示壳”升级为“可答辩、可演示、可持续扩展的原生医疗工作台”。
