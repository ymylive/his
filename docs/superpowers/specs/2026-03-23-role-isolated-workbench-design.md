# 轻量级 HIS 角色隔离专属工作台设计

## 1. 目标

在现有原生桌面端基础上，将当前“共享页面 + 角色裁剪”的工作台，重构为 `7 个角色完全隔离` 的专属工作台体系。

本轮设计目标：

- 每个角色登录后进入 `自己的首页`
- 每个角色拥有 `自己的左侧导航`
- 每个角色只看到 `自己的业务页面和快捷入口`
- 管理员不再作为万能入口，不直接进入一线业务操作流
- 患者端明确转为 `自助服务台`
- 住院登记员与病区管理员拆为两套独立工作台

## 2. 范围

本设计覆盖以下角色：

- 系统管理员
- 挂号员
- 医生
- 患者
- 住院登记员
- 病区管理员
- 药房人员

本轮设计覆盖：

- 页面架构
- 角色导航模型
- 角色首页结构
- 视觉层级
- 代码拆分
- 权限边界
- 测试与验证方案

本轮设计不覆盖：

- 新增业务能力
- 改写底层 service/repository 契约
- 多窗口、多弹窗、多进程桌面架构

## 3. 设计结论

### 3.1 总体方案

采用 `共享桌面外壳 + 7 个独立角色工作台模块` 的方案。

保留公共桌面基础能力：

- 登录页
- 顶栏
- 消息条
- 主题系统
- 公共控件
- 统一业务适配层

在此基础上，将角色工作台独立为单独模块，每个模块拥有：

- 独立首页
- 独立导航
- 独立页面集合
- 独立状态结构
- 独立快捷操作区

不采用以下方案：

- 每个角色维护一套完全独立的桌面应用壳
- 继续沿用现有共享页面并只做入口裁剪

### 3.2 设计原则

- 角色隔离靠结构保证，不靠页面内部条件分支兜底
- 先显示“当前角色今天要做什么”，再显示功能清单
- 一个角色只暴露与本职责直接相关的页面
- 当前 `DesktopAdapters` 继续作为统一业务适配层，避免 UI 重构演变为业务层重构
- 在 `raylib + raygui` 约束下保持单窗口、左侧导航、右侧工作区结构

## 4. 页面架构

### 4.1 共享外壳层

共享外壳负责：

- 登录成功后的角色工作台装载
- 顶部状态栏
- 全局消息条
- 主体布局
- 当前角色页面切换
- 退出登录

共享外壳不直接负责具体业务页面绘制。

### 4.2 角色工作台层

每个角色拥有独立工作台定义，包含：

- 角色标识
- 默认首页
- 左侧导航项
- 页面标题
- 强调色
- 页面渲染函数表

登录后不再进入一个“大工作台”再判断角色，而是直接挂载当前角色工作台。

## 6. 角色首页设计

所有角色首页统一采用如下区域骨架：

- 顶部角色标题区
- 首屏 `4 个核心指标卡`
- `今日优先任务`
- `快捷操作`
- `最近记录 / 风险提醒 / 待处理事项`

所有首页遵循同一布局规则，但内容完全按角色职责定制。

## 7. 视觉方向

### 7.1 视觉语言

采用 `统一医疗后台基座 + 角色专属强调色` 的视觉系统。

统一部分：

- 单窗口左侧导航
- 顶部状态栏
- 轻面板式卡片
- 信息优先级明确的工作区
- 高对比状态提示条

区分部分：

- 角色标题文案
- 角色首页卡片优先级
- 左侧导航命名
- 强调色

### 7.2 角色强调色建议

- 管理员：深海军蓝
- 挂号员：清晰青蓝
- 医生：专业青绿
- 患者：柔和天蓝
- 住院登记员：琥珀橙
- 病区管理员：靛青蓝
- 药房：草本绿色

### 7.3 视觉规则

- 首屏优先展示“今日任务”，不是功能列表堆叠
- 快捷操作使用大卡片，不使用密集小按钮
- 风险信息使用高对比提醒卡
- 最近记录区可数据密集，但不抢主任务视觉权重
- 左侧导航维持精简，单角色控制在 `3 到 6` 个入口

## 9. 交互流程

### 9.1 登录流

1. 用户输入账号、密码、角色
2. 系统校验账号密码
3. 系统校验账号真实角色与所选角色一致
4. 成功后进入该角色首页
5. 失败则停留在登录页并提示原因

不采用“角色不一致时自动兜底切换角色”的策略。

### 9.2 导航流

- 左侧导航只加载当前角色工作台页面
- 页面跳转在当前角色工作台内完成
- 如果由于异常状态跳入非本角色页面：
  - 自动回到本角色首页
  - 消息条提示无权访问

### 9.3 页面行为原则

- 操作失败保留表单输入
- 数据为空显示语义化空状态
- 成功操作后按角色首页需要刷新统计与最近记录
- 不因页面局部失败强制跳回首页

## 10. 权限边界

### 10.1 管理员

可看：

- 全局统计
- 主数据
- 汇总记录
- 审计与系统状态

可做：

- 维护患者、医生、科室、药品基础数据
- 查看汇总与审计

不可做：

- 直接执行挂号、接诊、发药、入院、转床等一线操作

### 10.2 挂号员

可看：

- 患者基本信息
- 挂号相关信息

可做：

- 建档
- 修改患者信息
- 办理挂号
- 查询或取消挂号

### 10.3 医生

可看：

- 本人待诊
- 患者历史
- 检查与诊疗信息
- 药品库存查询结果

可做：

- 写看诊记录
- 开检查
- 回写检查结果

不可做：

- 建患者
- 发药
- 入院登记
- 病区调度

### 10.4 患者

只可看：

- 本人挂号
- 本人看诊
- 本人检查
- 本人住院
- 本人发药

只可做：

- 自助查询
- 查看提醒

### 10.5 住院登记员

可看：

- 住院办理所需状态
- 床位可用性结果

可做：

- 入院
- 出院
- 住院状态查询

### 10.6 病区管理员

可看：

- 病房
- 床位
- 在床患者
- 出院前检查信息

可做：

- 查看病房
- 查询床位状态
- 转床
- 出院前检查

不可做：

- 正式办理入院或出院登记

### 10.7 药房

可看：

- 药品
- 库存
- 低库存
- 发药记录

可做：

- 新增药品
- 入库
- 发药
- 查询库存

## 11. 角色专属约束

- 医生登录后，`doctor_id` 自动绑定本人，不允许切换到其他医生视角
- 患者登录后，所有个人查询默认绑定本人 `patient_id`
- 药房登录后，`pharmacist_id` 自动绑定本人
- 管理员首页不出现一线业务办理入口
- 住院登记员与病区管理员工作台功能完全分离

## 12. 验证方案

### 12.1 状态层验证

- 登录后进入正确角色首页
- 登出后清空工作台状态
- 非本角色页面会被重定向

### 12.2 导航层验证

- 每个角色只显示自己的导航项
- 快捷入口只跳本角色页面
- 管理员不出现一线操作入口

### 12.3 权限层验证

- 患者只能看到本人数据
- 医生自动绑定本人视角
- 药房自动绑定本人发药人信息
- 非法跨角色访问返回失败提示

### 12.4 工作流层验证

- 挂号员接待流
- 医生接诊流
- 患者自助查询流
- 住院登记员入出院流
- 病区管理员转床与出院前检查流
- 药房入库发药流

## 13. 分阶段落地建议

### 阶段一

- 建立工作台注册层
- 拆分角色首页
- 角色导航独立化
- 登录落位改为强角色匹配

### 阶段二

- 将现有医生、患者、住院、药房页面拆入各自模块
- 新增挂号员、管理员、病区管理员专属首页

### 阶段三

- 补齐角色页面守卫
- 补齐角色专属测试
- 统一角色视觉层级

## 14. 风险与约束

- 当前 `DesktopPages.c` 体量较大，拆分时需避免多角色同时写同一文件
- 登录角色校验逻辑会改变现有“自动兜底角色”的行为，需要同步修正测试
- 在 `raylib + raygui` 下应避免过度复杂的多层嵌套交互
- 本轮重构以结构隔离为优先，不以花哨视觉为目标

## 15. 结论

本设计通过：

- 角色工作台独立建模
- 角色首页职责化
- 导航与页面访问强隔离
- 角色数据与行为绑定

将当前桌面端重构为真正的 `角色隔离专属工作台`。

该方案兼容现有桌面基础设施，能够在控制风险的前提下逐步推进实现。
