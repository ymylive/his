# 专门角色界面的医院一体化桌面工作台设计

## 1. 背景

当前项目已经具备较完整的 HIS 业务能力与桌面端雏形：

- 业务层已覆盖患者、挂号、接诊、检查、住院、病房床位、药房、账号认证等核心能力
- 桌面端已具备登录、首页、患者、挂号、医生、住院、药房、发药记录等基础页面
- 七类角色账号已经存在，且多数能力可通过 `MenuApplication / service / repository` 调用

但当前桌面端仍存在三类根问题：

- 显示层问题：中文字体加载与字形覆盖不稳定，出现乱码、缺字、方块字
- 布局层问题：窗口允许缩放，但页面大量使用固定像素定位，导致按钮和按钮、按钮和输入框、按钮和标题区发生重叠
- 业务层接线问题：多个角色页面能看不能办，或动作完成后不联动刷新，无法形成真正的医院一体化工作流

用户目标不是单点修补，而是把系统提升为：

- 七个角色拥有各自清晰的专门界面
- 同一条患者业务可以在多角色之间完整流转
- 页面在常用桌面分辨率下可操作、可演示、可答辩
- 配套一套中等规模、链路完整的模拟数据

## 2. 目标与边界

### 2.1 目标

- 交付统一桌面外壳下的七角色专门工作台
- 修复中文乱码、按钮重叠、主要元素遮挡、长文本溢出等前端显示问题
- 打通患者发起、挂号员接待、医生接诊、住院登记、病区管理、药房发药、管理员总览的跨角色主链路
- 构建答辩型模拟数据，保证七角色首页和三条完整业务链都有可演示内容

### 2.2 边界

- 保持现有技术栈：`C + raylib + raygui + CMake`
- 保持现有业务分层：桌面层通过 `DesktopAdapters -> MenuApplication -> service/repository` 调用
- 不引入收费、缴费、结算相关能力
- 不引入 Web 前端、不改为多窗口架构
- 不推翻现有后端服务层，只补公开应用接口、桌面适配接口与桌面工作台逻辑

### 2.3 非目标

- 真实号源排班系统
- 收费与结算流程
- 电子处方执行闭环
- 高级 BI 图表大屏
- 多窗口、多标签页复杂桌面框架

## 3. 方案比较与结论

### 3.1 方案 A：补丁式修复

做法：

- 在现有各角色页面上局部补按钮、补查询、补刷新、修按钮重叠、修字体

优点：

- 改动最小
- 见效最快

缺点：

- 继续沿用当前散乱页面结构
- 七个角色仍容易退化为“同页换标题”
- 无法稳定达到目标态工作台

### 3.2 方案 B：统一外壳 + 专门角色工作台重整

做法：

- 保留现有技术栈与业务层
- 统一桌面壳、主题、布局 helper、消息与刷新模型
- 为七个角色分别建设专门工作台
- 逐步补齐缺失的桌面接线与公开应用接口

优点：

- 可以同时解决显示、结构、业务闭环和数据演示问题
- 风险显著低于桌面近重写
- 与现有仓库结构最兼容

缺点：

- 改动范围较大
- 需要严格按波次交付，避免回归失控

### 3.3 方案 C：桌面层近重写

做法：

- 几乎重写整个桌面 UI 层，完全按目标态工作台重新组织

优点：

- 最接近理想状态

缺点：

- 风险最高
- 回归面最大
- 与当前用户目标相比性价比不高

### 3.4 结论

采用 **方案 B：统一外壳 + 专门角色工作台重整**。

## 4. 总体架构

系统保持单窗口桌面架构，但拆成四个明确层次：

### 4.1 Desktop Shell

职责：

- 登录后角色落位
- 顶栏、左侧导航、全局消息条
- 页面 freshness 状态：`clean / dirty / stale`
- 全局最小窗口约束与主工作区尺寸计算

主要承载：

- `DesktopApp`
- 顶层路由与导航

### 4.2 Theme And Layout

职责：

- 统一颜色、字体、间距、圆角、状态色
- 中文字体发现与字形加载
- 卡片、按钮组、信息行、列表/详情/操作区的共享布局 helper
- 面向 `1600x960` 主验收与 `1366x768` 最小支持分辨率的响应式规则

主要承载：

- `DesktopTheme`
- `WorkbenchCommon`
- 后续可抽出的共享 widgets

### 4.3 Desktop Adapters

职责：

- 桌面层唯一业务桥接点
- 聚合 DTO
- 统一 `Result / error_kind / snapshot_policy`
- 不缓存页面状态，不直接做页面导航

主要承载：

- `DesktopAdapters`
- `MenuApplication` 公开能力补齐

### 4.4 Role Workbenches

职责：

- 每个角色维护自己的页面状态、选中对象、筛选输入、上次成功快照
- 每个角色围绕本岗位主任务组织首页和子页
- 页面级重载只刷新依赖数据，不清空整个上下文

主要承载：

- `PatientWorkbench`
- `ClerkWorkbench`
- `DoctorWorkbench`
- `InpatientWorkbench`
- `WardWorkbench`
- `PharmacyWorkbench`
- `AdminWorkbench`

## 6. 一整套完整业务流程

系统必须支持同一名患者在多角色之间流转，至少形成以下三条完整可演示链路。

### 6.1 门诊到发药链

1. 患者自助挂号
2. 医生接诊
3. 药房发药
4. 患者查看发药记录与药品说明

### 6.2 门诊到检查链

1. 患者挂号
2. 医生开检查
3. 检查结果回写
4. 患者查看检查状态与结果

### 6.3 门诊到住院链

1. 患者挂号
2. 医生接诊并建议住院
3. 住院登记员办理入院
4. 病区管理员查看床位并可执行转床/出院前检查
5. 住院登记员办理出院
6. 患者查看住院历史

此外还必须支持两条角色侧链路：

- 挂号员可完成建档、挂号、查询、取消
- 管理员能在总览与管理页看到前述流程产生的数据变化

## 7. 页面结构与实现主战场

### 7.1 当前主战场

当前登录后的主渲染链主要位于：

- `DesktopTheme`
- `DesktopApp`
- `WorkbenchCommon`
- `workbench/*.c`

旧的 `DesktopPages` 仍然保留登录页、顶栏与遗留业务页，但登录后业务主路径应逐步收拢到角色 workbench。

### 7.2 必须优先治理的问题

#### 中文乱码

根因：

- 字体发现路径过窄
- 找不到字体时直接退回默认字体
- 已加载字体的字形覆盖过于依赖静态 UI 文案，运行时业务文本覆盖不足

设计要求：

- 扩大字体发现策略
- 明确字体回退顺序
- 将字形覆盖扩展到稳定支持本系统常见中文业务文本

#### 按钮与元素重叠

根因：

- 窗口可缩放，但子控件仍靠固定像素排布
- 父容器按比例分栏，子元素最小宽度不随容器变化
- 按钮区没有统一的换行/堆叠规则

设计要求：

- 同一操作区按钮必须有固定最小间距
- 当宽度不足时，按钮组必须自动换行或改为纵向堆叠
- 按钮不得压住按钮、输入框、标题或消息条

#### 长文本溢出

根因：

- 输出面板、信息行、摘要区对长文本缺少换行、裁切或滚动策略

设计要求：

- 长文本必须至少具备换行、裁切或滚动中的一种处理方式
- 不允许直接冲出容器边界

### 7.3 关键文件与职责

- `DesktopTheme.*`：字体、主题、最小窗口、基础视觉规则
- `DesktopApp.*`：角色落位、全局 freshness、导航和全局消息
- `WorkbenchCommon.*`：共享卡片、按钮组、列表详情壳、信息行、空态与错态
- `PatientWorkbench.*`：患者流程首页与自助挂号
- `ClerkWorkbench.*`：接待与挂号办理
- `DoctorWorkbench.*`：待诊、病史、接诊与检查
- `InpatientWorkbench.*`：入出院事务
- `WardWorkbench.*`：病房、床位、转床和出院前检查
- `PharmacyWorkbench.*`：库存、入库、发药和低库存提醒
- `AdminWorkbench.*`：跨域总览与维护入口
- `DesktopAdapters.*`：桌面业务桥接 DTO
- `MenuApplication.*`：公开应用接口补齐

## 8. 模拟数据设计

模拟数据以“中等规模、链路完整、适合答辩演示”为目标。

### 8.1 基础主数据

- 科室：不少于 `5`
- 医生：不少于 `10`
- 病房类型：不少于 `3`
- 床位：不少于 `12`
- 药品：不少于 `15`
- 角色账号：七类角色完整保留

### 8.2 核心剧情患者

建议提供 `8~12` 个核心患者，每人承担一类剧情：

- 可自助挂号
- 已挂号待诊
- 已接诊待检查
- 检查已回写
- 已住院
- 已发药
- 已出院
- 有取消挂号历史

### 8.3 背景运营数据

为了让列表和总览不显空洞，应额外补充：

- `60~100` 名患者
- 一定量历史挂号、检查、住院、发药记录
- 低库存和床位占用差异数据

## 9. 刷新联动模型

统一采用三态 freshness：

- `clean`：当前数据已同步
- `dirty`：依赖数据变更，页面应自动重载
- `stale`：重载失败，但保留上一次成功快照并给出警告

### 9.1 关键动作失效矩阵

#### 患者挂号成功后

标记为 `dirty`：

- 患者首页
- 患者挂号页
- 挂号员接待页
- 管理员挂号摘要

#### 挂号取消成功后

标记为 `dirty`：

- 患者首页
- 患者挂号页
- 挂号员挂号查询

#### 接诊/检查动作成功后

标记为 `dirty`：

- 医生待诊页
- 患者病历检查页
- 患者首页状态

#### 入院/出院/转床成功后

标记为 `dirty`：

- 住院登记员页面
- 病区管理员页面
- 患者住院状态页
- 管理员住院摘要

#### 发药/入库成功后

标记为 `dirty`：

- 药房工作台
- 患者发药用药页
- 管理员药品总览

### 9.2 页面刷新体验

- 成功动作后优先保留当前输入和当前选中对象
- 自动重载失败时保留上次成功快照，不允许白屏
- 消息条只显示最近一次关键反馈
- 长结果进入详情区，不塞进消息条

## 10. 必要的公开接口补齐

桌面层不应绕过 `MenuApplication` 直接操作 service/repository，因此必须补齐对桌面工作台有直接价值的公开应用接口。

### 10.1 需要优先补齐的接口类别

- 患者自助挂号的患者会话包装接口
- 科室列表与按科室列出医生接口
- 医生工作台需要的待诊与检查组合接口
- 管理员总览与各管理域的查询接口
- 病区管理员的转床与出院前检查接口
- 药品说明/用药方法查询接口

### 10.2 设计原则

- 优先包装已有业务能力，不重写核心业务规则
- 对患者相关写操作必须保留会话绑定与 ownership 约束
- `DesktopAdapters` 只做 DTO 组装，不拥有页面状态

## 11. 测试与验收标准

### 11.1 自动验证

至少通过：

- `cmake --build build`
- `ctest --test-dir build --output-on-failure`

测试覆盖必须包含：

- 七角色默认首页与状态切换
- `dirty / stale / clean` 状态流
- 患者、医生、住院、药房、管理员关键 DTO
- 三条完整工作流
- 床位冲突、库存不足、重复挂号、权限隔离等边界条件

### 11.2 界面硬门槛

验收分辨率：

- 主验收：`1600x960`
- 最小支持：`1366x768`

在两种分辨率下，七角色主流程页必须满足：

- 中文稳定显示
- 按钮不与按钮重叠
- 按钮不压输入框
- 标题、消息条、列表、详情区不互相覆盖
- 长文本不冲出容器
- 主操作按钮完整可点击
- 不出现影响操作的横向裁切
- 空状态必须有说明和下一步提示

### 11.3 流程通过标准

必须能人工走通：

- 门诊到发药链
- 门诊到检查链
- 门诊到住院链
- 挂号员建档/挂号/取消链
- 管理员跨域查看链

### 11.4 最终通过条件

只有同时满足下面条件，才算本轮完成：

- 七个角色都有专门界面
- 七个角色至少各有一条可操作主流程
- 三条跨角色完整链路可演示
- 模拟数据足够支撑答辩，无需临时造数
- `1600x960` 与 `1366x768` 下关键页面无按钮重叠和主要元素遮挡
- 中文显示稳定
- 构建、测试、桌面 smoke 全部通过

## 12. 实施波次约束

虽然本设计覆盖七角色目标态，但实现必须分波次推进，避免一次性改动失控。

### 12.1 团队并行开发约束

实现阶段必须采用团队并行开发，而不是单线程串行推进。

约束如下：

- 主智能体负责统一计划、分配任务边界、汇总结果、处理冲突与最终集成
- 子智能体按工作流并行拆分，至少覆盖：
  - 基础治理与共享布局
  - 患者/挂号员/医生门诊链
  - 住院/病区链
  - 药房/管理员链
  - 测试与验收
- 所有并行开发子智能体统一使用 `gpt-5.4`
- 任务拆分必须避免多个子智能体同时修改同一文件，优先按共享层、角色层、测试层切分
- 每个子智能体任务必须明确：
  - 负责范围
  - 完成条件
  - 验证命令
- 主智能体不得把最终集成与全量验证外包；最终冲突裁决与集成验收必须由主智能体完成

建议波次：

### Wave 1：基础治理

- 字体系统
- 统一主题
- `WorkbenchCommon` 响应式布局
- 最小窗口约束
- 按钮/输入/消息条不重叠

### Wave 2：患者、挂号员、医生

- 患者主链路
- 挂号员接待闭环
- 医生待诊与接诊检查闭环

### Wave 3：住院、病区、药房、管理员

- 住院登记员

## 13. 结论

本设计选择在保留现有技术栈和业务层的前提下，重整桌面工作台为“统一外壳 + 七角色专门界面 + 跨角色完整链路 + 答辩型模拟数据”的目标态。

它不是简单补按钮，也不是推倒重写，而是一次有边界的系统升级：

- 先修字体、布局、响应式这类基础问题
- 再补桌面接线与公开应用接口
- 最后把七个角色真正组织成一套能演示、能联动、能答辩的医院一体化桌面系统
